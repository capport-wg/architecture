Updates

  • Component Diagram: distinguish API from Web portal
  • Attempting to improve precision of language

Principles

  • No man-in-middle of DNS, HTTP, etc.
  • Use Internet Protocol (vs. access-specific)
  • Notify on any protocol, not just port-80 HTTP
  • Machine-readable notifications, not (only) human eyeballs on web page
  • Backwards compabibility, incremental deployment
  • Facilitate trust mechanisms

Protocols in capport-architecture

  • Notification (ICMP): replacement for HTTP or DNS modification
  • API: because we don't want the URI in the notification
  • Provisioning: to learn of the API
  • Web interface: to fulfil requirements of portal

Components in capport-architecture

  • Enforcement: Blocking access to wider network, sending ICMP
  • API Server: Providing Information
  • DHCP server, or gateway: Boot-strapping network and URI to API
  • Web server: User interface to navigate portal

Enforcement

  • Currently, this function blocks most traffic (except within walled garden) and modifies port-80 HTTP or DNS
  • We propose the new Enforcement function sends a form of ICMP "unreachable" message.
  • This tangibly improves the reaction to non-port-80-HTTP
  • HTTP modification of http port-80 may be used for some time

API

  • Expected to be an idempotent RESTful API
  • Basic:
    • Indicates captivity
    • Indicates web interface URL
  • Advanced:
    • Remaining bytes/time quota

  • Tangible improvement over MITM HTTP modification
